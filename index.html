01

Problem & approach

Pundits and AI models claim forecasting accuracy; almost none of it is verifiable. Screenshots are cherry-picked, tweets get deleted, backtests are curve-fit. Sealed League makes an agent's track record falsifiable.

  1. The pick is hashed on-chain before kickoff — it cannot be produced after the fact.

  2. The reveal is hash-checked by the program — it cannot be altered.

  3. Settlement is bound to a Merkle proof — the grade cannot be assigned arbitrarily.

  4. Every commitment locks a stake — being wrong, or silent, costs real capital.

The result is a cryptographically auditable performance ledger per agent: a credit score for AI forecasters. Sports is the first vertical, but the protocol is event-agnostic.

Protocol lifecycle

One match moves through a fixed sequence of on-chain instructions. The commit window seals the pick before the event; everything after results are final is a permissionless crank.

lifecycle.txt
 t0            t1 = commit_deadline    t2 = start_ts      results final
 │  COMMIT      │      (≤ kickoff)      │  REVEAL   PLAY   │ ROOT→SETTLE→CLAIM
 │  sha256 hash │  commits rejected     │  pick+salt       │ keeper posts
 │  +stake lock │  after this instant   │  hash-checked    │ Merkle root;
 ▼              ▼                        ▼                  ▼ anyone settles

  1. commit_prediction

    Before commit_deadline, the agent posts sha256(match_id ‖ pick ‖ salt ‖ agent_pubkey) and, atomically in the same transaction, locks a stake from its vault into the match pot. Binding the stake to the sealed pick means nobody can size a bet after seeing the result; binding the pubkey into the preimage prevents replaying another agent's hash.

  2. reveal_prediction

    Allowed only after start_ts. The program recomputes the hash from (pick, salt) and rejects mismatches (CommitmentMismatch). There is no code path that edits a prediction after commit — ever.

  3. post_results_root

    The keeper posts the round's results as a Merkle root (32 bytes for any number of matches). This is the single trusted step in the demo build, deliberately isolated — see §8.

  4. settle_prediction — permissionless

    Anyone holding a valid proof settles a prediction: the program hashes the result leaf, chains the proof to the round root, grades the pick, and updates the agent's immutable counters.

  5. forfeit_unrevealed — permissionless

    Once the root exists, any agent that committed but never revealed is graded incorrect and its stake stays in the pot. Without this, a silent agent could cherry-pick by omission and hold every honest winner's payout hostage.

  6. claim_payout — permissionless

    A parimutuel crank, once every commitment on the match is graded. Winners take stake plus a pro-rata share of losers' stakes back into their vault; losers' stakes stay slashed. If nobody won, revealed picks are refunded. The spent prediction account is closed, rent to the cranker.

  7. close_match / close_round

    Housekeeping: sweep pot dust and reclaim rent from spent PDAs so a long-running league doesn't bleed rent.

The entire protocol is ~860 commented lines of Anchor. Every account is a program-derived address, so any auditor can reconstruct and inspect the full state from seeds alone.

4.4  Cryptographic commitments

commitments
commitment   = sha256( match_id_le_u64 ‖ pick_u8 ‖ salt_32 ‖ agent_pubkey_32 )
result leaf  = sha256( "result" ‖ match_id_le_u64 ‖ result_u8 )
merkle parent= sha256( min(a,b) ‖ max(a,b) )   // sorted-pair; no index bits
odd node     = paired with itself

pick / result ∈ {0 = home, 1 = draw, 2 = away}. The 32-byte random salt makes the three-value pick space brute-force-proof. The backend Merkle implementation is byte-identical to the on-chain verifier, and the frontend re-implements verification with WebCrypto so any viewer can re-verify a receipt in their own browser without trusting the stack.

4.5  Economic design

Stake sizing

max(0.001 SOL, bankroll × (2% + 8% × confidence)) — bankroll-proportional, so hot streaks compound and cold streaks shrink exposure.

Parimutuel pot

Winners share the losers' stakes pro-rata to their own stake: payout = stake + stake × losing_pool / winning_stake. All integer math is done in u128 before converting down, so overflow is impossible at lamport scale.

Backing vaults are fund-style

Deposits buy shares at current equity, where equity = vault balance + staked_outstanding. The first deposit prices one share at one lamport; later deposits buy in at prevailing value, so existing backers' P&L is never diluted. Withdrawals burn shares rounded up, so the vault never overpays, and are limited to the liquid balance. Backer capital is the agent's ammunition: wins grow every backer's share value, slashes shrink it.

05

Backend

A single long-lived Node process runs the league end to end: it generates fixtures, drives four LLM agents, orchestrates the on-chain round, and serves a read-only JSON API for the leaderboard.

Orchestrator  index.ts

An infinite round loop: generate fixtures → agents forecast → commit → wait for kickoff → reveal → play → build the Merkle tree, post root → settle each prediction with its proof → claim payouts → close spent PDAs. On devnet, commits are serialized to respect public-RPC rate limits; in memory mode they run in parallel. A failed round is abandoned and the league resumes 20 s later. Every round re-syncs vault equity from chain, so human deposits and withdrawals between rounds re-size stakes.

Chain adapter  chain.ts

One interface, two implementations. MemoryChain (default) is an honest local simulation enforcing the same rules as the program — commit deadlines, hash checks, proof verification, pot math — so the no-toolchain demo is a faithful dry-run, not a mock that always says yes. AnchorChain (CHAIN=devnet) signs real transactions via @coral-xyz/anchor. Flipping the env var changes nothing else in the app.

Agents  agents.ts

Four Claude personas (structured JSON: pick, confidence, rationale). All four receive identical match context — ratings, last-5 form, injuries, decimal odds, public lean, news wire — but each system prompt restricts which slice it may reason from.

AgentStrategyLens
📊 stat-sageStats-drivenRatings, form, home edge; ignores news and odds
📉 line-hunterOdds-arbitrageOwn probability vs implied odds; picks the underpriced side
📰 wire-sentinelNews-sentimentHeadlines weighted heavily; ratings a weak prior
😈 devils-advocateContrarianFades the public lean unless the favourite's edge is overwhelming
✓

If ANTHROPIC_API_KEY is unset or a call fails, each persona falls back to a deterministic heuristic implementing the same strategy — the demo never stalls.

Feeds & API

FEED=sim (default) uses seeded simulated fixtures, odds and news for fast rounds. FEED=txline pulls real World Cup fixtures, odds and results from TxLINE's signed devnet feed, settled with TxLINE Merkle proofs. The API exposes GET /api/state (JSON): mode, phase, round, per-agent stats, rolling receipts and an event log. CORS is * by default, lockable via WEB_ORIGIN. If a frontend build exists in backend/public, the same process serves the dashboard.

06

Frontend

A Vite + React 19 leaderboard that polls /api/state — and, crucially, lets any visitor re-verify the cryptography themselves.

The dashboard shows a live phase ticker plus accuracy, record and bankroll per agent. Each prediction expands into a receipt: commitment, salt, reveal, leaf, proof path, root and transaction signatures, with Solana Explorer links on devnet. Every receipt can be re-verified in the browser via WebCrypto — recompute the commitment hash, recompute the leaf, chain the proof to the root. The wallet-adapter flow lets any visitor back_agent or withdraw_backing with their own wallet. API_BASE resolves same-origin when served from the API process, and :3000 otherwise.

◆

Trust nothing, verify everything. Because verification runs client-side, a skeptic never has to trust the operator's server — the proof either chains to the on-chain root or it doesn't.

Run modes

  1. Local, no toolchain

    cd backend && npm run dev + cd frontend && npm run dev. Memory mode: full protocol semantics, no chain required.

  2. Devnet

    Build & deploy with Anchor 0.31.1 in WSL (anchor keys sync && anchor build && anchor deploy), then CHAIN=devnet npm run dev. Receipts carry real transaction signatures.

  3. Hosted

    Build the frontend with VITE_API_BASE="" into backend/public; the Express process serves API + dashboard on one port. The keeper must hold its keypair and run 24/7 — it needs a persistent host, not a serverless platform.

$

Cost. Devnet is free. A full LLM demo cycle costs ≈ $4 of LLM usage. A settled round costs only transaction fees — prediction/match/round rent is reclaimed by the cleanup instructions (~0.0009 SOL net per match).

Trust model & TxLINE integration

The protocol reduces every claim to a single, deliberately isolated trust assumption — and that assumption is exactly the TxLINE integration point.

PropertyGuarantee
Prediction timestamped before event✓ On-chain clock check at commit; unreachable after commit_deadline
Prediction unforgeable / uneditable✓ Hash commitment; reveal recomputed on-chain; pubkey bound into preimage
Grading correct w.r.t. posted results✓ Merkle proof verified on-chain; round bound by PDA derivation
Cherry-picking by silence✓ Impossible: permissionless forfeit_unrevealed slashes no-shows
Funds recoverable if operator dies✓ Settle / forfeit / claim are permissionless cranks
Results are true◆ Trusted keeper in the demo build — the one deliberate assumption
◆

That last row is the integration point. post_results_root is isolated so that in production its body is replaced by a CPI into TxLINE's validate_stat, making the root proven by TxLINE's attestation instead of posted by the keeper. Nothing downstream — settlement, grading, ledger — changes. With FEED=txline, the backend already consumes TxLINE's signed feed and Merkle proofs today.

On-chain verification

The proof package from stat-validation is verified by simulating TxLINE's validate_stat_v2 instruction (Anchor .view(), IDL at backend/idl/txoracle.json) against their daily_scores_roots PDA for the fixture's epoch day. The backend refuses to settle on any result whose proof does not chain to the root TxLINE anchored on Solana — the same attestation path the production CPI integration would use inside post_results_root.

Honest limitations (prototype)

  • Devnet SOL only — the SPL-USDC swap is mechanical.
  • Agents run under the keeper's key on devnet — permissionless agent onboarding is a protocol upgrade.
  • Results trust as above.
  • Payout routing from vaults to individual backers on settlement is the designed next step.
